# Real-Time Fraud Detection Data Platform

A production-style fintech data platform that simulates how companies detect fraudulent transactions at scale using PySpark pipelines, behavioral risk scoring, and an interactive monitoring dashboard.

## 1. Project Objective

This platform replicates how a real-world fraud monitoring system operates in fintech environments:

- Process high-volume transaction data through layered pipelines
- Detect behavioral anomalies such as velocity spikes, device switching, and location changes
- Balance fraud detection accuracy with false positives
- Provide near real-time visibility to fraud analysts through a monitoring dashboard

## 2. High-Level Flow

1. Synthetic transactions are generated or uploaded as CSV.
2. Bronze layer ingests raw records.
3. Silver layer validates, normalizes, and quarantines bad records.
4. Gold layer computes features, risk score, and fraud alerts.
5. Streamlit dashboard reads Gold outputs and renders KPI and analytics views.

4. Layer Responsibilities
Bronze Layer (Raw Ingestion)

Module: src/layers/bronze_layer.py

Reads latest uploaded CSV or synthetic data

Stores raw transactions without transformation

Writes transactions_bronze.parquet

Silver Layer (Cleaning and Validation)

Module: src/layers/silver_layer.py

Deduplicates using transaction_id

Normalizes timestamps and schema

Drops null-critical records

Quarantines invalid rows (nulls, non-positive amounts)

Writes transactions_silver.parquet and quarantine.parquet

Gold Layer (Features, Scoring, Alerts)

Module: src/layers/gold_layer.py

Builds behavioral fraud features

Computes risk scores and assigns risk levels

Generates alerts based on thresholds and anomaly rules

Writes:

features_gold.parquet

scored_gold.parquet

fraud_alerts.parquet

pipeline_metadata.parquet

5. Synthetic Data Engine

File: src/ingestion/data_generator.py

Key behavior:

No deterministic seeding to ensure variability

Unique identifiers per record (user_id, merchant_id, device_id, transaction_id)

Random dataset size within realistic range

Indian names and locations for contextual realism

Fraud pattern distribution:

normal: 98.5%

card_testing: 0.7%

device_switch: 0.4%

location_anomaly: 0.3%

high_value: 0.1%

Expected fraud rate: approximately 0.2% to 1%

6. Risk Scoring Model

File: src/fraud_rules/risk_scoring.py

Risk formula:

risk_score = 0.4 * transaction_velocity
           + 0.3 * device_switch_frequency
           + 0.3 * location_change_rate

Risk levels:

LOW if risk score < 0.3

MEDIUM if 0.3 <= risk score < 0.7

HIGH if risk score >= 0.7

7. Alert Rules

File: src/fraud_rules/rule_engine.py

Alerts are generated for:

All MEDIUM and HIGH risk transactions

Velocity anomaly (>10 transactions within a short time window)

High-value transaction from a first-seen device

Multiple location changes within one hour

10. Common Questions
Why do I see part-* and _SUCCESS files?

These are standard Spark output artifacts:

part-*: parquet partitions generated by distributed tasks

_SUCCESS: job completion marker

They are expected in runtime directories and should not be committed.

Why is there no Python code inside data/bronze, data/silver, data/gold?

Because this follows production design:

src/ contains pipeline logic

data/ contains pipeline outputs

This separation improves maintainability and scalability.
